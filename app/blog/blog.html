What is MVC?

Simply put, MVC is a method for organizing code to make it scalable. Specifically, here are the three parts of MVC:

Model – The model is essentially the data used to populate the application. For example, logic that is related to a database would be the model component of an MVC app.

View – The view is what the user sees. For example, HTML files that display in a web browser would be the view component of an MVC app.

Controller – The controller is the bridge between the model and the view. It takes input/requests from the user interacting with the view and finds what is needed from the model to serve back to the user.

What kind of scope do variables have?

Variables can have global or local scope. Variables with global scope can be accessed from anywhere, and variables with local scope can only be accessed from within its context.

Special Cases

A variable without the “var” in front of it is automatically global. A nested function can access variables in its parent function, but the parent function cannot access variables in the nested function.

Closures allow variables in an outer, but not global, scope to be “remembered.”

Git Workflow for Team Projects

One of the major focuses of any project is using git and github properly while having the right workflow. This post is a step-by-step guide to having a proper git/github workflow for team/group coding projects.

Step 1: Create a central github repository

Create a central remote repository for your project. For some of my projects, we created a new organization, which is a feature that Github allows. Then, we each joined this organization to contribute to this central repository.

Step 2: Each team member forks and clones the central repository

Make sure you do not clone from the central repository directly. Each team member will be pushing to their own remote repository and then making pull requests to the central repository for review before getting their changes merged in.

Make sure your changes are specifically related to this new branch that you have just created, to keep things organized. Otherwise, it will be difficult for team members to figure out what exactly you completed in your pull request/commits.
